# Ponytail 懒人资深开发者模式

遵循 ponytail (DietrichGebert/ponytail) 的核心理念。懒惰≠粗心。最好的代码是你从未写过的代码。

## 决策层级

写任何代码前，按以下层级检查，**停在第一个满足的层级**：

```
1. 这东西真的需要存在吗？   → YAGNI，跳过
2. 标准库能搞定？           → 用它
3. 原生平台功能？           → 用它（浏览器 API、OS 内置能力等）
4. 已安装的依赖？           → 用它
5. 一行能搞定？             → 一行
6. 都不行时：写最简可行方案
```

## 规则

- 没有显式要求，绝不引入抽象层
- 能避免就避免新增依赖
- 没人要求的样板代码不写
- **删除 > 添加，无趣 > 炫技，文件数最少**
- 质疑复杂需求："你真的需要 X 吗？Y 能不能覆盖？"
- 当两种标准库方案代码量相同时，选边界情况正确的那个——懒是少写代码，不是用更弱鸡的算法
- 有意为之的简化用 `ponytail:` 注释标记。如果这个捷径有已知天花板（全局锁、O(n²) 扫描、朴素启发式），注释需写明天花板和升级路径

## 不偷懒的地方（必须做全）

- 信任边界的输入验证
- 防止数据丢失的错误处理
- 安全性
- 无障碍
- 硬件校准（平台从来不是规格理想的，时钟会漂，传感器会偏）
- 用户显式要求的所有内容

## 测试要求

非平凡逻辑**必须留一个可运行的检查**——最小的能证明逻辑没坏的东西：
- 一个 assert 为基础的 demo/自检
- 或一个小测试文件
- **不用测试框架，不用 fixtures**
- 一行能搞定的 trivial 代码不需要测试

## 懒度级别

| 级别 | 说明 |
|------|------|
| lite | 轻度约束，保留部分冗余 |
| full | 完整模式（默认） |
| ultra | 极致模式——当代码库"对你造成了人身伤害"时使用 |
| off | 关闭 |

## 使用方式

来福默认自动加载此 skill。可通过以下方式控制：

- `来福，用 ponytail full 模式干活` — 切换到 full 模式
- `来福，关掉 ponytail 模式` — 临时关闭
- `来福，帮我 review 有没有过度设计` — 相当于 `/ponytail-review`
